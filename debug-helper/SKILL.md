# Debug Helper

## 触发条件

用户报告 bug、错误信息、崩溃、异常行为，或明确要求调试帮助时激活。

## 调试流程

### Step 1: 收集信息
- 错误消息/堆栈跟踪
- 复现步骤（若用户未提供，主动询问）
- 环境信息（OS、运行时版本、依赖版本）
- 最近的代码变更（使用 git log/diff）

### Step 2: 定位根因
- 从错误堆栈反向追踪调用链
- 使用搜索工具找到相关源码
- 识别失败点的上下文

### Step 3: 分析并输出

## 输出模板

```markdown
## 问题诊断

**错误类型**: [编译错误 / 运行时错误 / 逻辑错误 / 性能问题]
**影响范围**: [受影响的模块/功能]

## 根因分析
[清晰描述问题的根本原因，而非表面症状]

## 调用链
[展示从入口到失败点的调用路径]

## 修复方案

### 方案 1（推荐）
- 修改说明：[...]
- 具体代码变更：[...]

### 方案 2（备选）
- 修改说明：[...]

## 验证步骤
1. [如何验证修复是否生效]
2. [建议添加的回归测试]
```

## 调试技巧

- **编译错误**：检查头文件包含、类型匹配、语法错误
- **运行时崩溃**：检查空指针、越界访问、资源泄漏
- **逻辑错误**：对比预期输出与实际输出，定位分叉点
- **并发问题**：检查竞态条件、死锁、数据竞争
- **性能问题**：检查热路径中的不必要操作、内存分配

## 规则

- 使用中文回答，专业术语保留英文原文
- 给出修复方案时直接提供可运行的代码
- 如果问题不确定，明确告知用户并说明需要哪些额外信息
- 优先推荐最小化改动的修复方案
